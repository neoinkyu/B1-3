# 프로젝트 1. 만족도 조사 후속 대응 자동화 도구 비교

## 1. 프로젝트 개요

본 프로젝트는 Google Form으로 접수된 만족도 조사 결과를 기준으로 응답을 자동 분류하고, 분류 결과를 Google Sheets에 기록한 뒤 응답자에게 맞춤형 이메일을 발송하는 워크플로우를 구현한 결과물이다.

동일한 업무 흐름을 **Make**와 **Activepieces**에서 각각 구현하여 Trigger, Action, 조건 분기의 작동 방식을 확인하고, 두 도구의 사용성과 적합한 활용 상황을 비교하였다.

- 자동화 대상 업무: 만족도 조사 결과 확인 및 후속 안내
- 사용 도구: Make, Activepieces
- 공통 연동 서비스: Google Forms, Google Sheets, Gmail
- 보너스 과제: 미수행

---

## 2. 자동화 업무 정의

### 기존 수작업

1. 설문 응답이 접수되었는지 확인한다.
2. 만족도 점수를 확인한다.
3. 2점 이하 응답은 개선이 필요한 의견으로 별도 정리한다.
4. 3점 이상 응답은 일반 응답으로 별도 정리한다.
5. 응답 점수에 맞는 이메일을 작성하여 발송한다.

### 자동화 목적

- 설문 접수 이후의 반복 확인 업무를 줄인다.
- 만족도 점수에 따라 응답을 일관된 기준으로 분류한다.
- 분류 결과와 후속 이메일 발송 이력을 자동으로 남긴다.
- 동일한 워크플로우를 서로 다른 도구에서 구현하여 도구별 차이를 비교한다.

## 3. 공통 워크플로우 설계

```text
Google Form 제출
        ↓
Google Sheets에 새 응답 행 생성
        ↓
조건 분기
 ├─ 만족도 2점 이하: 개선 필요
 │    ├─ 개선요청 시트에 행 추가
 │    └─ 개선 의견 접수 이메일 발송
 │
 └─ 만족도 3점 이상: 일반 응답
      ├─ 일반응답 시트에 행 추가
      └─ 감사 이메일 발송
```

### Trigger와 Action

- **Trigger**: Google Sheets의 설문응답 탭에 새 행이 추가됨
- **조건 분기**
  - 개선 필요: 만족도 2점 이하
  - 일반 응답: 만족도 3점 이상
- **Action 1**: 분기 결과에 해당하는 Google Sheets 탭에 행 추가
- **Action 2**: 응답자에게 분기별 이메일 발송

### 조건 분기 기준

| 구분 | 조건 | 결과 시트 | 이메일 유형 |
|---|---:|---|---|
| 개선 필요 | 만족도 2점 이하 | 개선요청 | 의견 접수 및 개선 검토 안내 |
| 일반 응답 | 만족도 3점 이상 | 일반응답 | 설문 참여 감사 안내 |

Activepieces의 일반 응답 조건은 화면에서 `만족도 > 2`로 설정하였다. 설문 점수가 1~5점의 정수이므로 Make의 `만족도 3점 이상` 조건과 동일하게 작동한다.

## 6. Make 구현

### 6.1 구현 과정 요약

1. `Google Sheets - Watch New Rows`를 Trigger로 설정하였다.
2. Router를 추가하여 만족도 점수에 따라 두 경로로 나누었다.
3. 각 경로에 Filter를 적용하였다.
4. `Google Sheets - Add a Row`로 결과 탭에 데이터를 기록하였다.
5. `Gmail - Send an Email`로 분기별 이메일을 발송하였다.
6. 저점수와 고점수 데이터를 각각 실행하여 두 경로가 정상 작동하는지 확인하였다.
7. 일정 실행을 활성화하여 자동 실행 구조를 구성하였다.

## 7. Activepieces 구현

### 7.1 구현 과정 요약

1. `Google Sheets - New Row Added`를 Trigger로 설정하였다.
2. Router를 추가하여 개선 필요, 일반 응답, Otherwise 경로를 구성하였다.
3. 개선 필요 경로는 만족도 2점 이하로 설정하였다.
4. 일반 응답 경로는 만족도 2점 초과로 설정하였다.
5. 각 경로에서 `Google Sheets - Add Row`로 결과 탭에 데이터를 기록하였다.
6. 각 경로에서 `Gmail - Send Email`로 맞춤형 이메일을 발송하였다.
7. Flow를 Publish한 뒤 저점수와 고점수 테스트를 각각 실행하였다.

## 9. 도구 비교 분석

| 비교 항목 | Make | Activepieces |
|---|---|---|
| UI/UX | 원형 모듈을 연결하는 시각적 캔버스 구조로 전체 흐름을 한눈에 확인하기 쉽다. | 단계가 위에서 아래로 배열되고 Router가 가지 형태로 표시되어 순서 파악이 직관적이다. |
| 초기 설정 난이도 | Search Method, 시작 위치, Filter 연산자 등 세부 설정 항목이 많아 초기에 익숙해지는 시간이 필요했다. | Trigger 테스트 후 값을 선택하는 흐름이 비교적 단순했으나 Publish 개념을 별도로 이해해야 했다. |
| 조건 분기 | Router와 각 경로의 Filter를 별도로 설정한다. 경로별 조건을 세밀하게 제어하기 좋다. | Router 안에서 Branch 조건을 직접 설정하고 Otherwise 경로를 기본 제공한다. |
| 데이터 매핑 | 각 모듈의 필드를 후속 모듈 입력란에 시각적으로 삽입할 수 있으며 세부 데이터 확인이 편리하다. | 테스트된 이전 단계의 값을 선택하여 매핑한다. Trigger 샘플 데이터가 준비되어야 후속 설정이 수월하다. |
| 실행 테스트 | Run once와 모듈별 실행 버블을 통해 어느 단계까지 처리됐는지 바로 확인할 수 있다. | 각 단계의 Test 기능과 Runs 화면을 이용해 단계별 성공 여부를 확인한다. |
| 실행 로그 | 캔버스의 실행 버블과 실행 이력에서 입력·출력을 자세히 볼 수 있어 문제 지점 추적이 쉽다. | Runs 화면에서 실행 경로와 단계별 결과를 확인할 수 있으며 구조가 비교적 간결하다. |
| 자동 실행 방식 | Scenario를 활성화하고 실행 주기를 설정한다. Polling Trigger의 실행 간격을 이해할 필요가 있다. | Flow를 Publish하면 설정한 Trigger를 기준으로 실행된다. Draft와 Published 상태가 구분된다. |
| 워크플로우 가독성 | 복잡한 자동화와 다중 분기를 한 화면에 시각화하는 데 강점이 있다. | 비교적 단순한 흐름은 읽기 쉽지만, 단계가 많아지면 세로 길이가 길어질 수 있다. |
| 학습 곡선 | 기능이 많고 설정이 세밀한 만큼 초보자에게는 다소 어렵지만 확장성이 높다. | 기본 자동화는 빠르게 익힐 수 있으나 일부 용어와 Publish 방식에 적응이 필요하다. |
| 본 과제 적합성 | 조건 분기와 실행 결과를 시각적으로 설명하기 좋아 비교 과제에 적합했다. | 동일한 구조를 간결하게 재현할 수 있어 두 번째 비교 도구로 적합했다. |

---

## 10. 도구별 장점과 단점

### 10.1 Make

#### 장점

- 전체 워크플로우가 시각적으로 잘 드러난다.
- Router와 Filter를 이용한 복잡한 조건 분기에 강하다.
- 각 모듈의 입력·출력 데이터를 상세하게 확인할 수 있다.
- 실행 오류가 발생한 위치를 찾기 쉽다.
- 다양한 모듈을 추가하여 후속 자동화를 확장하기 좋다.

#### 단점

- 초기에 설정해야 할 항목이 많다.
- Polling Trigger의 실행 시점과 시작 위치 개념이 처음에는 혼동될 수 있다.
- 모듈과 분기가 많아지면 화면 배치가 복잡해질 수 있다.
- 세부 기능이 많은 만큼 간단한 자동화에도 설정 과정이 길게 느껴질 수 있다.

### 10.2 Activepieces

#### 장점

- Trigger, Router, Action의 순서가 위에서 아래로 명확하게 보인다.
- Branch와 Otherwise 구조가 직관적이다.
- 단순한 조건 분기 자동화를 비교적 빠르게 구성할 수 있다.
- 테스트된 데이터를 기준으로 후속 필드를 선택할 수 있어 매핑 흐름이 명확하다.
- Flow Publish 상태를 통해 편집본과 실행본을 구분할 수 있다.

#### 단점

- Trigger 샘플 데이터가 준비되지 않으면 후속 매핑이 어렵다.
- Draft 수정 후 Publish를 누르지 않으면 실제 실행에 변경사항이 반영되지 않는다.
- 긴 워크플로우는 화면이 세로로 길어질 수 있다.
- Make와 비교하면 실행 캔버스에서 세부 데이터가 즉시 드러나는 정도는 상대적으로 낮았다.

## 11. 상황별 적합한 도구

### Make가 더 적합한 경우

- 여러 앱과 데이터를 복합적으로 연결해야 하는 경우
- Router와 Filter가 여러 단계로 이어지는 복잡한 업무
- 실행 과정의 입력·출력을 자세히 확인해야 하는 경우
- 워크플로우를 시각적으로 설계하고 팀에 설명해야 하는 경우

### Activepieces가 더 적합한 경우

- Trigger와 몇 개의 Action으로 구성된 비교적 단순한 자동화
- 위에서 아래로 순차적인 흐름을 선호하는 경우
- Branch와 Otherwise를 이용해 분기 구조를 빠르게 구성하려는 경우
- 복잡한 설정을 최소화하고 빠르게 프로토타입을 만들고 싶은 경우

### 최종 의견

이번 만족도 조사 후속 대응처럼 두 개의 조건으로 분기하고 시트 기록과 이메일을 발송하는 업무는 두 도구 모두 충분히 처리할 수 있었다.

다만 실행 과정을 상세히 확인하고 향후 조건이나 Action을 더 확장해야 한다면 Make가 유리하다. 반대로 구조가 단순하고 빠른 구현이 중요하다면 Activepieces가 더 편리하다.

이번 프로젝트에서는 **Make가 전체 흐름과 실행 상태를 시각적으로 확인하는 데 더 강점이 있었고**, **Activepieces는 동일한 자동화를 비교적 간결하게 재현하는 데 강점이 있었다.**

## 12. 구현 중 확인한 문제와 해결 과정

Make 테스트 과정에서 Google Form 응답은 `설문응답` 탭에 정상 입력되었지만, 처음에는 `Make_일반응답` 탭에 새 행이 생성되지 않았다.

원인을 확인하기 위해 다음 순서로 점검하였다.

1. Trigger가 새 행을 정상적으로 수신했는지 확인
2. 만족도 값이 일반응답 Filter 조건을 통과했는지 확인
3. Add a Row의 대상 스프레드시트와 탭 확인
4. 각 결과 열의 데이터 매핑 확인
5. 새로운 테스트 응답으로 재실행

설정을 점검하고 수정한 뒤 일반응답 시트에 행이 정상적으로 생성되는 것을 확인하였다. 이 과정을 통해 자동화 오류가 발생했을 때 Trigger, Filter, Action 순서로 실행 지점을 확인하는 것이 효과적이라는 점을 배웠다.

## 15. 결론

본 프로젝트를 통해 동일한 만족도 조사 후속 대응 업무를 Make와 Activepieces에서 각각 구현하였다.
